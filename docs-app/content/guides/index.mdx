# Guides & Recipes

topics ships **five concepts** — [Topic, Record, seq, Router, Tombstone](/getting-started/mental-model) — plus a permanent delete operation, and a small `/v0` HTTP surface. These guides show how those primitives compose into the messaging patterns you actually deploy: a **job queue**, **pub/sub fan-out**, a **durable replayable stream**, and **N-way multi-master** mirroring. Each is a recipe — copy the `curl`, adapt the config, ship it.

The point of this section: topics is not four products. It is one append-only log, made durable per topic, with server-side fan-out and an explicit gap signal. The "patterns" are just configuration choices on that one substrate.

## The substrate, in one paragraph

A **topic** is a named append-only log. You `POST` records into it and read them back from a `from_seq` cursor with [`/diff`](/api/reading), or live over [SSE watch](/api/watch). What turns a plain topic into a queue, a pub/sub feed, or a durable stream is **config**: the [durability class](/guarantees/durability) (`memory` / `disk` / `fsync`), the retention caps (`cap_records` / `cap_bytes` / `ttl_ms`), the full-topic policy (`discard: "old"` vs `"reject"`), and whether it is a [lease queue](/api/queues) (`type: "queue"`). [Routers](/api/routers) copy one topic's records into another, at-least-once and per-source FIFO, preserving the origin `$node` so [loop-prevention](/guarantees/node-loop-prevention) makes mirroring echo-free.

<Callout type="info">
  Every recipe here is plain HTTP and JSON against `$TOPICS` (your server's base URL — see
  the [Quickstart](/getting-started/quickstart)). `curl` is a complete client — there is no
  SDK to install. The examples assume the dev-mode default (loopback bind, auth disabled);
  see [Security](/deployment/security) for the production setup.
</Callout>

## Which guide do I need?

- **Many producers, many workers, work must not be lost, each job processed once-ish.** You want a [Job Queue](/guides/job-queue) — either a cursor queue (Bull-style) or a lease queue.
- **One publisher, many subscribers, low latency, bounded memory, gaps tolerable.** You want [Pub/Sub](/guides/pubsub).
- **A consumer must be able to replay from any point, and must never silently miss a record.** You want a [Durable Stream](/guides/durable-streams).
- **Several symmetric nodes that each write and each consume the union, without echo.** You want [Multi-Master Fan-out](/guides/multi-master).

    Two models: a Bull-style durable cursor queue, and a lease queue with

  <Cards.Card title="Pub/Sub" href="/guides/pubsub" arrow>
    Redis-style fan-out: a small capped+TTL feed, routers to per-subscriber topics,

  </Cards.Card>
  <Cards.Card title="Durable Streams & Replay" href="/guides/durable-streams" arrow>
    Strong delivery: unbounded + <code>fsync</code> + <code>discard:"reject"</code> so
    eviction is impossible and there is no tombstone source — consumers replay from their
    last acked cursor.
  </Cards.Card>
  <Cards.Card title="Multi-Master Fan-out" href="/guides/multi-master" arrow>
    N-way mirroring: each node sets its <code>node</code> id, symmetric routers between
    nodes, and node loop-prevention makes the topology echo-free by construction.
  </Cards.Card>
</Cards>
