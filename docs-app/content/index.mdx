# Millions of events a second, with the simplicity of <span className="grad">SQLite</span>

<p className="hero-sub">
topics is a persistent event log that runs as **one process on one machine** — no cluster,
no coordination, the way SQLite is one file instead of a database fleet. Yet the engine
moves **millions of events a second** over a small HTTP/JSON API. Write to a **topic**, read
it back by **sequence number**, follow it live, delete what you're done with — and choose,
per topic, how durable each write should be. Nothing is ever lost without telling you.
</p>

```bash
# Point $TOPICS at your server, then append an event. The topic is created
# on first write, and the server assigns the sequence number.
export TOPICS=http://localhost:4000
curl -X POST $TOPICS/v0/topics/orders \
  -H 'content-type: application/json' \
  -d '{ "records": [ { "data": { "sku": "AEROPRESS-GO", "qty": 1, "total": 3499 } } ] }'
# → { "topic": "orders", "seqs": [1], "head_seq": 1, ... }

# Read everything after sequence 0. The sequence number is the cursor.
curl -X POST $TOPICS/v0/topics/orders/diff -d '{ "from_seq": 0 }'
# → { "records": [ { "$seq": 1, "$ts": 1748470000123, "data": { "sku": "AEROPRESS-GO", ... } } ],
#     "next_from_seq": 1, "caught_up": true }
```

<section className="section">

<span className="section-eyebrow">The API</span>
<h2 className="section-title">The whole thing is a handful of HTTP calls</h2>
<p className="section-lede">
topics is a log you <code>POST</code> events to and read back by number. The server stamps
each event with a monotonic <code>seq</code>; reading is just "give me everything after this
number." That sentence is the entire model — here is every verb it expands into.
</p>

<div className="concept-grid">
  <div className="concept"><div className="c-name">create</div><p className="c-desc">A topic appears on first write — or <code>PUT /v0/topics/:topic</code> to set config up front. Nothing to provision.</p></div>
  <div className="concept"><div className="c-name">write</div><p className="c-desc"><code>POST /v0/topics/:topic</code> with a batch of records. The server numbers each one and returns the assigned seqs.</p></div>
  <div className="concept"><div className="c-name">read</div><p className="c-desc"><code>POST …/diff</code> with <code>from_seq</code>. Get the records after it plus the next cursor. The seq is the cursor — advancing it is the ack.</p></div>
  <div className="concept"><div className="c-name">watch</div><p className="c-desc"><code>POST /v0/watch</code>, then open the SSE stream. Follow many topics live over one resumable connection, woken on append.</p></div>
  <div className="concept"><div className="c-name">delete</div><p className="c-desc"><code>POST …/delete</code> by seq range and/or tag. Permanent, immediate, point-in-time — it never touches records written later.</p></div>
  <div className="concept"><div className="c-name">fan out</div><p className="c-desc"><code>PUT /v0/routers/:r</code> to copy every record from one topic into another, server-side, preserving origin.</p></div>
</div>

<p className="section-lede" style={{ marginTop: '1.4rem' }}>
No connection modes, no binary framing, no consumer-group bookkeeping on the server.
Read it at three levels of depth on the <a href={withBase("/api")}>API reference</a>.

<span className="section-eyebrow">Run it on one machine</span>
<h2 className="section-title">Like SQLite: huge capability, almost nothing to operate</h2>
<p className="section-lede">
SQLite became the most-deployed database on earth by refusing to be a server you run — it's
a library and a file. topics brings that operating model to event streaming: a Kafka-shaped
job done with an SQLite-shaped footprint. It <em>is</em> a networked server, so every service
can share it — but running it is one process and one directory, not a cluster to coordinate.
</p>

<section className="section">

<span className="section-eyebrow">Durability you choose</span>
<h2 className="section-title">Decide what an "ok" means — one topic at a time</h2>
<p className="section-lede">
Every write travels the same road: from your process, into the kernel, onto the physical
disk. The only question is how far along that road topics waits before it answers "ok." You
decide <strong>per topic</strong>, so a throwaway cache and a financial ledger can live side
by side in one process without paying for each other's guarantees.
</p>

<div className="dura-grid">
  <div className="dura">
    <span className="d-tag">memory</span>
    <h4>Fastest — no promise</h4>
    <p>Acked the instant topics accepts it, with <strong>no durability guarantee</strong>: after a restart its records may survive or may be gone. For caches, scratch, and ephemeral fan-out where losing a little is fine.</p>
  </div>
  <div className="dura">
    <span className="d-tag">disk</span>
    <h4>Survives a crash</h4>
    <p>Written to a group-committed write-ahead log before the ack. <strong>Survives a process crash</strong>, losing only the most recent not-yet-flushed tail. The everyday default.</p>
  </div>
  <div className="dura">
    <span className="d-tag">fsync</span>
    <h4>Survives power loss</h4>
    <p>The ack waits until the data is <strong>fsynced to the platter</strong>. An acknowledged write survives <em>any</em> crash, power loss included, and is replayed from the log on restart. For ledgers and queues.</p>
  </div>
</div>

<p className="section-lede" style={{ marginTop: '1.4rem' }}>
Whatever the class, an acked write is published and nothing visible is ever un-durable. Read
the full contract on <a href={withBase("/guarantees/durability")}>Durability Classes</a>.
</p>

<section className="section">

<span className="section-eyebrow">Mental model</span>
<h2 className="section-title">Five nouns and you know the system</h2>
<p className="section-lede">
topics is deliberately small. Learn five concepts and every endpoint is just one of them
doing its job.
</p>

<div className="concept-grid">
  <div className="concept"><div className="c-name">Topic</div><p className="c-desc">A named, append-only log of records ordered by a monotonic <code>seq</code>. Created lazily on first write.</p></div>
  <div className="concept"><div className="c-name">Record</div><p className="c-desc">One immutable event. The server assigns <code>$seq</code> and <code>$ts</code>; you supply opaque <code>data</code>, an optional <code>tag</code> and <code>node</code>.</p></div>
  <div className="concept"><div className="c-name">seq</div><p className="c-desc">The cursor. Reads are "give me everything after <code>from_seq</code>." The client owns its position; advancing it is the ack.</p></div>
  <div className="concept"><div className="c-name">Router</div><p className="c-desc">A server-side <code>source → dest</code> forwarding rule. Fan-out that preserves origin node, so N nodes mirror without echo.</p></div>
  <div className="concept"><div className="c-name">Tombstone</div><p className="c-desc">The explicit "you missed data" signal. Cap/TTL loss returns an in-band gap range — at HTTP 200, never silently.</p></div>
</div>

<p className="section-lede" style={{ marginTop: '1.4rem' }}>
The load-bearing rule connecting them: <strong>loss you didn't ask for is always announced;
removal you did ask for is silent.</strong> Read it in full in the
<a href={withBase("/getting-started/mental-model")}>mental model</a>.
</p>
