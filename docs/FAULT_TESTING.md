# streams — Fault-Injection & Crash-Consistency Test Suite

This documents how `streams` is tested against hostile filesystems and crashes, in the
tradition of SQLite's VFS fault tests and PostgreSQL's recovery/WAL-consistency checks. The
promise under test: **data loss is explicit, never silent** — an acked durable write survives any
crash; an unacked write may vanish but never creates a rule-violating gap.

> **Status — implemented.** The harness described below (`trait Fs` + `RealFs` / `FakeDisk` /
> `FaultFs` / `MonitorFs`, routed through wal/snapshot/segstore/recovery) is **built and in use**;
> the "Harness" / "Crate plan" sections are written in imperative/design tense but describe shipped
> code (`src/storage/fs.rs`, `src/storage/testfs.rs`). All frame/segment/snapshot checksums are
> **XXH3-64, not CRC32** — the "CRC" fault-name labels below (e.g. `F-WAL-CRC-FLIP-*`) are stable
> test identifiers; the checks they describe assert XXH3-64 mismatches.

## Harness

FS SEAM: introduce `trait Fs: Send+Sync { fn open(&self,path,opts)->Result<Box<dyn File>>; fn rename(&self,from,to); fn remove_file; fn read_dir; fn sync_dir(path); fn create_dir_all; }` and `trait File { fn read_at(&self,off,buf); fn write_at(&self,off,buf)->Result<usize/*short-write*/>; fn set_len(len); fn sync_data(); fn sync_all(); fn metadata_len(); }`. Refactor src/storage/wal.rs ActiveFile (open_for_append/write_all_at_tail/fdatasync/set_len), src/storage/snapshot.rs write_snapshot (tmp/sync_all/rename/fsync_dir), src/storage/segstore.rs LocalSegmentStore (write_atomic/put/delete/read_range/fsync_dir), and src/engine/recovery.rs (WalReader::open/truncate_active) to take an `Arc<dyn Fs>` instead of calling std::fs directly. Production wires RealFs (zero behavior change). 

THREE Fs IMPLS: (1) RealFs = thin std::fs wrapper for prod + the ground-truth real-disk crash job. (2) FakeDisk = in-memory `BTreeMap<PathBuf, FileImage>` where FileImage holds `durable: Vec<u8>` + `pending: Vec<(range,bytes)>`; write_at appends to pending; sync_data/sync_all promotes this file's pending->durable; sync_dir promotes pending directory-entry ops (create/rename/unlink) to durable; crash() drops all un-promoted pending bytes AND un-fsynced dir ops, optionally applying torn-write damage (truncate a pending append to a random prefix, zero-fill a sector, or garble) to the LAST pending write per file — deterministic from a seed. rename is modeled as a durable-only-after-sync_dir directory-entry change. (3) FaultFs = wraps either RealFs or FakeDisk and injects an error at a chosen global call index N: fail-once or fail-always, on a chosen op class (write_at/sync_data/sync_all/rename/sync_dir/set_len/read_at), returning EIO/ENOSPC/short-write/ESTALE.

MONITOR FS (always-on cross-check, SQLite journal-test analog): a passive wrapper that asserts ordering invariants live during ALL tests: (a) no WAL byte range is reported committed (CommitToken Ok) before that file's sync_data returned for a durable batch; (b) snapshot .tmp is never renamed before its sync_all; the old snapshot file is removed only after the new rename + sync_dir; (c) in LocalSegmentStore.put, .idx is never written before .data is durable, and sync_dir follows both; (d) cold relocation: cold.put fsync completed before confirm_relocated flips the tier pointer, and hot().delete happens only after the flip. A violation panics immediately, independent of crash tests.

EXHAUSTIVE CRASH-POINT LOOP (SQLite sweep): an op-under-test (e.g. one durable append batch, one snapshot write, one segment seal, one full relocation, or one recovery run) executes against FaultFs/FakeDisk while a counter records the TOTAL number of FS mutating calls M. Then `for crash_point in 0..M { reset disk to pre-op durable image; run op but crash() (drop pending + apply torn damage seed) exactly after the crash_point-th FS call; reopen via recover_and_open; assert the model-oracle invariants }`. Two semantics per sweep: fail-once (transient glitch then succeed — tests recovery-after-glitch) and fail-always (dead device/ENOSPC). For each crash_point also iterate a small set of torn-damage seeds (prefix-truncate / zero-sector / garble) so atomicity within a single write is exercised, ALICE-style.

MODEL ORACLE (PG wal_consistency_checking analog): a pure-Rust reference model is a `BTreeMap<box_id, Vec<(seq,Record)>>` + floors + tombstones + router set + delete set, updated by the SAME logical op stream but only committing an op once its ACK is observed (durable batch fsync returned). After every injected fault+recovery, dump the recovered engine state (head_seq/earliest_seq/count/per-seq data/tag/node/floors/tombstone-reason/router-edges/router-cursor) and diff against the model with the contract relaxation: recovered ⊆ model∪{acked} and acked ⊆ recovered (acked-durable must survive; unacked may be absent but must not create gaps). For the redo path specifically, optionally embed/recompute a per-frame post-state hash and compare on replay (fatal on divergence). 

IDEMPOTENT-RECOVERY CHECK: run recover_and_open TWICE (recovery(recovery(state)) must be byte-identical: same head/floors/records/segment registry) and also run recovery, capture, then run recovery again on the recovered dir. 

LAYERED TOOLS: proptest generates the op sequence + chooses the crash step + shrinks; FakeDisk.crash() decides survivors; fail-rs places the named crash-point precisely in prod code; loom/shuttle drive the concurrency-only tests on the commit-token/visibility primitives (cfg(loom)/cfg(shuttle) sync-module swap). One RealFs subprocess SIGKILL job (the existing tests/crash_recovery.rs pattern) stays as kernel-fidelity ground truth the fake cannot model.

## Crate plan (pure Rust — no RocksDB/redb/3p storage engine)

- proptest (already dev-dep) — backbone op-sequence generator + shrinking for the stateful crash harness; drive {append/delete/seal/roll/relocate/reclaim/snapshot/restart/crash-point} sequences and shrink to minimal repro
- proptest-state-machine 0.x — formalize the reference-model oracle layer (apply ops to a pure in-memory model, compare to recovered engine); optional but turns the ad-hoc oracle into a state machine
- loom 0.7 — exhaustive 2-thread model check of the commit-token/visibility boundary: CommitState{Mutex<CommitOutcome>+Condvar} handoff, head_seq Acquire/Release publish vs reader, WalWriter mpsc single-consumer drain. cfg(loom) sync-module swap
- shuttle 0.7 — randomized+replayable scheduler for the larger multi-actor assembly (N concurrent writers + WAL writer thread + background relocator + sealer + reader) that exceeds loom's state budget; replay saved schedule string on failure
- fail 0.5 (failpoints feature, off in release) — named crash-points/EIO at the exact persistence call sites (wal::after_write, wal::after_fdatasync, snapshot::after_rename, snapshot::before_dirfsync, segput::after_data_before_idx, cold::after_copy_before_flip, cold::after_flip_before_delete, recovery::mid_replay, recovery::before_truncate, reclaim::between_unlinks). It triggers WHEN to crash; FakeDisk decides WHAT survives
- serial_test 3 — serialize tests sharing the global failpoint registry (fail-rs footgun); or scope failpoints per-test with a registry guard
- xxhash-rust (already a dep, xxh3 feature) — reuse XXH3-64 inside FakeDisk to verify torn-write/bit-rot detection matches the engine's own frame checksum (XXH3-64, not CRC32)
- tempfile (already dev-dep) — per-test isolated data dirs for the RealFs ground-truth crash job
- arbitrary 1.x — derive structured op/fault generation for fuzz-style sequence input (feeds proptest or a cargo-fuzz target over the recovery decoder)
- OWN CODE (not a crate, highest leverage): an `Fs`/`File` trait abstracting std::fs (open/read_at/write_at/set_len/sync_data/sync_all/rename/remove_file/read_dir/File::sync), with RealFs (production), FaultFs (EIO/ENOSPC/short-write injection at call indices), and FakeDisk (in-memory page store with pending-vs-durable byte ranges, crash()=drop-pending, torn-write tearing at 512B/4KB granularity, rename+dir-fsync atomicity model). The WAL ActiveFile, snapshot writer, LocalSegmentStore, and recovery all route through this trait. NO rocksdb/redb/sled/3p storage engine — streams owns its WAL/segment/snapshot format
- NOTE: explicitly NOT adopting turmoil/unstable-fs (requires std::fs swap + async, API unstable, gives nothing FakeDisk doesn't); NOT adopting any third-party storage/KV engine

## Model-oracle invariants (checked after every injected fault + recovery)

1. DURABILITY (acked⇒durable): every write whose CommitToken returned Ok (durable batch fsync completed) is present after recovery with identical seq/ts/data/tag/node — acked ⊆ survivors. Checked by diffing recovered records against the model's acked set.
2. NO-SILENT-LOSS / NO-GAP: an unacked write may be absent, but the recovered seq set is always a contiguous prefix with no rule-violating hole between two surviving records — survivors form [base..head] minus only explicitly-deleted seqs (count==head-base+1-deleted). No surviving seq implies a missing earlier surviving seq.
3. SEQ MONOTONIC + CONTIGUOUS: head_seq never regresses across restart; recovered seqs are dense (the burst test's count==head invariant); a new post-recovery append continues at exactly head+1.
4. REPLAY IDEMPOTENT / CONVERGENT: recover_and_open run 1, 2, or 3 times yields byte-identical state (head/floors/records/segment registry/router cursor). WAL replay's Append seq-skip (seq<=head) means an overlapping checkpoint frame is never double-applied; a delete/evict-watermark replays to the same floor.
5. TORN-TAIL TRUNCATION, NEVER MISREAD: a torn/partial/garbled WAL frame (bad frame_len overrun, sub-minimum frame_len, internal-length inconsistency, XXH3-64 checksum mismatch, non-utf8 node/tag, unknown type) is the logical end of log — recovery truncates at the last checksum-valid frame's end and never materializes a bogus record. valid_len equals the offset of the last good frame.
6. CORRUPTION EXPLICIT, NEVER SERVED: a sealed segment .data/.idx frame failing XXH3 surfaces SegmentError::Corrupt (not silent truncation, since sealed segments are immutable); a torn/checksum-bad snapshot is skipped (load_latest falls back to an older valid snapshot or full WAL replay), never decoded as state. Bit-rot is detected at every read boundary.
7. ATOMIC SNAPSHOT SWAP: after a crash anywhere in tmp-write→fsync→rename→dir-fsync→prune-old, recovery loads exactly one consistent snapshot — either the new one (if rename+dir-fsync durable) or the previous valid one — never a half-written body and never zero snapshots when a valid older one existed (old is pruned only after new is durable).
8. SEGMENT .idx⇒.data CONSISTENCY: a recovered segment's .idx never references a .data byte range that isn't durably present; a crash between the .data and .idx put leaves a segment that recovery treats as incomplete (LocalSegmentStore.list requires the .data part; the WAL/snapshot remains the source of truth so no live record is lost).
9. COLD RELOCATION ALL-OR-NOTHING: after a crash at any of copy→fsync→flip-tier-pointer→delete-hot, exactly one readable copy of every segment exists (never zero); BoxTier::resolve prefers HOT during the both-exist window; the hot copy is deleted only after the (durably re-derivable) cold copy exists; the relocator re-runs the idempotent copy/flip/drop and loses nothing.
10. DELETE NEVER RESURRECTS: a seq removed by a replayed Delete (selector or explicit seqs) is absent after recovery and stays absent across repeated recovery; a reclaimed/orphan segment whose unlink was interrupted is dropped on the next boot (reclaim_segments_on_recovery) and never resurfaces.
11. FLOORS PRESERVED + MONOTONIC: evict_floor/expiry_floor/delete_floor and delete_below survive crash and never regress — an involuntary cap/TTL tombstone still tombstones a below-floor cursor after restart; a voluntary delete stays silent (no tombstone) after restart.
12. LEASE-LOG PROJECTION: with leases_durable:true, replayed Lease events rebuild the who-holds-what projection deterministically and a crash never double-acks/double-claims a job; with leases_durable:false nothing replays and every in-flight job is claimable again (self-healing) with no lost or duplicated queue record.
13. IDEMPOTENCY WINDOW IS BEST-EFFORT, NEVER UNSAFE: the dedupe map is not persisted, so after a crash a retried idempotency_key may re-execute (allowed) but must never produce a gap, never assign a duplicate seq to a live record, and never resurrect a deleted seq — the retry is a fresh contiguous append at head+1 or a no-op.
14. VISIBILITY/PUBLISH ATOMICITY (concurrency): a reader gating on head_seq (Acquire) that observes head==N always finds slots [base..N] populated (publish stores records before the Release of head_seq); a staged-but-unpublished batch is invisible; a rollback_staged after a failed fsync leaves the box byte-identical to pre-stage and publishes nothing. No reader ever observes a record that is not yet durable for a durable box.
15. COMMIT-TOKEN CORRECTNESS (loom): the CommitState handoff resolves every submission exactly once to Ok (after the group fsync for a durable frame) or Failed (writer io error/gone); no waiter blocks forever, no lost wakeup, no torn/future seq observed across the single mpsc writer drain.
