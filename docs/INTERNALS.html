How topics persists data & survives disk failure

A top-to-bottom tour of the storage engine: the path a write takes from an HTTP request down to fsynced bytes, the on-disk formats that hold it, and the exact mechanics by which a crash, a torn write, or a bad disk sector loses nothing it promised to keep.

00The one-line mental model

topics is a persistent event engine. You POST records into a named topic (a log or a queue) and read them back in order, durably, across restarts. Everything about its disk story collapses to a single sentence:

The durability boundary

The Write-Ahead Log (WAL) is the one and only line between "remembered" and "forgotten." A write is durable exactly when its frame is in the WAL on disk. Everything else on disk — segments, snapshots, indexes — is a performance-and-serving optimization that can be rebuilt by replaying the WAL. Only data not yet in the WAL can be lost.

Hold that thought through every layer below. The segments make reads fast; the snapshots make restart fast; the indexes make lookups fast. None of them are the source of truth. The WAL is.

01Layered architecture

From the socket down to the platter, a write passes through five layers. Each one has a single job and a clean seam to the next.

① HTTP / wire layer  src/http/, src/serve.rs
axum 0.8 over hyper 1.10. Parses the request, authenticates the API key (SHA-256 hashed, constant-time compare, scope bits), enforces global limits (topic/router/watch/inflight caps → 429), hands a typed RecordIn to the engine.
② Engine / write orchestration  src/engine/mod.rs
The 11-step write: validate → resolve/auto-create the topic → idempotency dedupe → admission control (retention, caps, byte quota) → assign seqs under the append lock → enqueue to the WAL → wait for durability → forward to routers → publish & wake watchers.
③ In-memory state  topic_state.rs, eviction.rs, queue.rs, router.rs
A DashMap of topics; each topic holds an index (VecDeque<RecordLoc> + tag BTreeMap), an atomic head_seq as the visibility watermark, retention floors, and an optional queue projection.
④ Durable storage  storage/wal.rs, segment.rs, snapshot.rs
The WAL (truth), segments (materialized per-topic data + index), and metadata snapshots (checkpointed state). All framed and XXH3-checksummed; all written through a narrow filesystem trait.
⑤ Filesystem seam  storage/fs.rs
A single injectable Fs / File trait that all durability I/O routes through. Production uses RealFs (a thin std::fs wrapper); tests swap in hostile in-memory disks to drive crashes and I/O errors precisely.
Why a library crate?

The binary (main.rs) is a thin wrapper. Exposing the engine as a library lets the integration suite drive the exact same in-process app via tower::ServiceExt::oneshot — no sockets, no sleeps, no flakiness. The crash tests run the real engine against a fake disk.

02The write path, top to bottom

Here is what actually happens when a record is written. The ordering is load-bearing — it is the reason a crash at any instant leaves a consistent state.

 HTTP POST /topics/{name}/records
        │
        ▼
 ┌─────────────────────────────────────────────────────────────────────┐
 │ 1. validate        size limits (data+meta ≤1 MiB, tag ≤256B, …)      │
 │ 2. resolve topic     existing, or auto-create from config              │
 │ 3. idempotency     dedupe by client key inside the idempotency window│

 ├─────────────────────────────────────────────────────────────────────┤

 │ 6 ║ stage_append   assign contiguous seqs (head+1, head+2, …)    ║    │
 │ 7 ║ wal_enqueue    push frame(s) onto the WAL writer's channel   ║    │
 │ 8 ║ ticket         take next_publish_ticket  ──►  DROP THE LOCK  ║    │
 │   ╚═════════════════════════════════════════════════════════════╝    │
 ├─────────────────────────────────────────────────────────────────────┤
 │ 9. durability gate  wait on the CommitToken  (OFF the lock!)          │
 │10. forward          fan out to routers (at-least-once)               │
 │11. publish          advance head_seq · wake watchers · record dedupe │
 └─────────────────────────────────────────────────────────────────────┘

The subtlety is in steps 6–9. Sequence assignment and WAL enqueue happen under a per-topic lock so seqs are dense and the WAL order matches the seq order. But the expensive part — waiting for fsync — happens after the lock is dropped, so many concurrent writers' fsync waits coalesce into one disk flush. A ticket taken under the lock guarantees writers still publish in strict seq order even though they wait in parallel. (More in §3.3.)

2.1 — Three durability classes

Not every topic wants the same trade between speed and safety. Each topic picks a Durability class that decides how far down the stack a write must reach before the engine answers 200 OK.

ClassWhat "ack" meansSurvives a crash?Cost
memory Takes the same group-committed WAL write path as disk (a frame is written); fully queryable. Best-effort — no guarantee. After a restart its records may survive or be lost; the topic is never reset-to-empty by contract and head_seq never regresses above the acked head. Config always survives. Never fsync-gated. Fastest.
disk Frame is in the WAL's OS write buffer (group-committed), not necessarily fsynced. Process crash: yes. Power loss / kernel panic: the un-fsynced tail may be lost — but never silently (see tombstones). One batched write(). No per-write fsync.
fsync Frame is fsynced to stable storage — the platter, not just the page cache. Yes — any crash, including power loss. One fdatasync per batch, amortized across the group.
The clever part

Even disk writes go through the same group-commit machinery as fsync writes — they just don't gate the ack on the fdatasync. A single batch can mix both: the writer issues exactly one fdatasync if any record in the batch is fsync-class, and only the fsync-class waiters are released after it completes. The disk-class waiters are released as soon as the buffered write() returns.

03The Write-Ahead Log

The WAL is an append-only sequence of length-prefixed, checksummed frames in files named wal-<idx>.log. It is the engine's heartbeat and its source of truth. Three properties make it trustworthy: every frame is self-describing and checksummed, writes are group-committed by a single thread, and a failed write never leaves bytes behind.

3.1 — Frame format

Each record becomes one frame. The header is fixed at 30 bytes; the trailing checksum is 8 bytes of XXH3-64. Multi-byte integers are little-endian.

 off  size  field          note
 ───  ────  ─────────────  ─────────────────────────────────────────────
   0    4   frame_len  u32   bytes of this frame EXCLUDING this prefix
   4    1   type       u8   1=Append 2=TopicCreate … 6=Delete … 10=Lease 11=HeadWatermark
   5    1   flags      u8   bit0=has_tag bit1=has_node bit2=durable
   6    4   topic_id     u32
  10    8   seq        u64   the record's monotonic sequence number
  18    8   ts         u64   server commit time (ms)
  26    2   node_len   u16
  28    2   tag_len    u16
  30    4   data_len   u32
  34    N   node       bytes
   .    M   tag        bytes
   .    P   data+meta  bytes   ← opaque payload envelope
   .    8   xxh3       u64   XXH3-64 over bytes [4 .. crc_start)

Two design choices here do all the crash-safety work:

  • frame_len comes first. A reader can validate a frame boundary — "do I even have this many bytes?" — before trying to parse the body. A half-written frame at the end of the file fails this check immediately.
  • XXH3-64 trailer over the whole body. If the length looks right but a byte is torn or rotted, the checksum mismatches. The reader treats that as the end of valid data.
Why XXH3, not CRC32

The on-disk integrity hash is xxHash XXH3-64 everywhere — WAL frames, segment frames, snapshot bodies. It is dramatically faster than CRC32 on modern CPUs (multi-GB/s) while giving a wider 64-bit check field, and the same function is reused across all three formats so there is one crash anchor to reason about.

The torn-tail-safe reader

On startup, a WalReader iterates frames and stops at the first frame that fails the length or checksum test. It reports a valid_len() — the byte offset of the last good frame's end. Everything past that is a torn write from a crash mid-append, and recovery truncates the file to exactly that length. The reader cannot be fooled into reading a partial frame as a real one.

3.2 — Single-writer group commit

All WAL writes funnel through one dedicated OS thread. Producers (the async request handlers) submit a record over an mpsc channel and get back a CommitToken they can wait on. The writer thread runs a tight loop:

 WriterTask::run()  — the group-commit loop
 ─────────────────────────────────────────────────────────────
 loop {
   batch = [ recv_timeout(50ms) ]          // block for the first record
   drain_ready(batch)                       // try_recv everything queued NOW

   if batch has any fsync-class record:
       adaptive_window(batch)               // linger 500µs..10ms to coalesce
       drain_ready(batch)                   // pick up what arrived during the wait

   len_before = active.len
   for rec in batch: active.write_frame(rec)   // one buffered write per frame

   if any record was fsync-class:
       active.fdatasync()                   // ONE flush for the whole batch

   publish_outcomes(batch)                  // release every waiter as Ok
 }
gc_min
500 µs — the minimum linger to let a batch form
gc_max
10 ms — the cap on how long a write will wait to coalesce
file_size

The adaptive window only triggers for durable batches: there is no point lingering to coalesce fsyncs if nothing in the batch needs an fsync. Under load, hundreds of writes collapse into a single fdatasync — that is where the throughput comes from.

3.3 — Off-lock fsync (the ~8× win)

The naïve design holds the per-topic append lock across the fsync wait, so writers serialize on the slowest part of the path. The current design splits the critical section:

 Writer A          Writer B          Writer C
 ────────          ────────          ────────
 │ lock                                          ┐
 │ seq=5                                          │  Under the lock: ONLY
 │ enqueue                                        │  seq assignment +
 │ ticket=5                                       │  WAL enqueue + ticket.
 │ unlock ──────►  lock                           │  Microseconds.
                   │ seq=6                         ┘
                   │ enqueue
                   │ ticket=6
                   │ unlock ──────►  lock
                                     │ seq=7 …
 │ wait fsync ◄──── coalesced ────► │ wait fsync   ┐  OFF the lock: all three
 │                                  │              │  fsync waits land on the
 │ ◄──── one fdatasync releases all ─────────►     │  SAME batch flush.
 │ publish 5  (ticket gate)         │             ┘
                   │ publish 6  (only after 5)
                                     │ publish 7

Because seq assignment is still serialized, the WAL order equals the seq order. Because the fsync wait is not serialized, the flushes coalesce. And because each writer took a monotonic ticket under the lock, the publish step is gated in strict ticket order — writer 6 cannot make its record visible until writer 5 has. So readers still see a perfectly ordered, gap-free stream even though the durability waits happened concurrently.

Invariant — publish atomicity

A record becomes visible (head_seq advances to cover it) only after its durability class is satisfied and all lower-seq records have published. There is no window in which a reader can observe seq N but not seq N−1, nor observe an un-durable record from a durable topic.

3.4 — Tail-rewind on error

What if the write() or the fdatasync() fails partway through a batch — a full disk (ENOSPC), an I/O error (EIO)? The writer captured len_before at the top of the batch. On any write or fsync error it rewinds the active file's logical length back to len_before and fails every waiter in the batch.

No half-written batch ever persists

Failed, un-acked bytes are truncated away before any acknowledgement is sent. A writer either gets a clean Ok (its frame is fully on disk to its durability class) or a Failed (its frame is not on disk and the file is back to its pre-batch length). There is no third outcome where a frame is acked but only partly written.

Two more details close the loop on WAL durability:

  • Parent-dir fsync before first ack. Creating wal-0.log isn't durable until the directory entry is fsynced. The WAL fsyncs the parent directory before acknowledging any write into a newly created file — otherwise a crash could lose the file itself, not just its contents.
  • Rotation names from the file index, not a counter. The next WAL file is wal-<idx+1>.log derived from the current file's real index, so a crash during rotation can never produce two files claiming the same index or a gap.
  • Drain-and-fsync on shutdown. The writer's Drop path drains the channel and fsyncs, so a clean shutdown loses nothing even for disk-class topics.

04Segments & tiering

The WAL is great for writing but bad for reading old data: it interleaves every topic's records and grows without bound. So committed Append records are materialized per topic into segment files — the long-term, queryable form. This is purely a serving optimization; segments are always reconstructible from the WAL.

4.1 — The .data / .idx pair

A segment covers a contiguous seq range [start_seq, end_seq] and is two files that sort by seq because the start-seq is zero-padded in the name:

seg-<start>.data
append-ordered framed records — one XXH3-checksummed frame each
seg-<start>.idx
fixed 20-byte stride locator — entry i ⇔ seq start+i

The .data frame is a close cousin of the WAL frame, minus the type byte — a segment only ever holds Append records, so the type is implied. Header is 25 bytes:

 .data frame                          .idx entry  (20-byte stride)
 ───────────────────────────          ──────────────────────────────
  0  4  frame_len  u32                 0  4  offset  u32  → frame in .data
  4  1  flags      u8                  4  4  len     u32  → framed length
  5  8  seq        u64                 8  8  ts      u64  → inline, for TTL math
 13  8  ts         u64                16  1  flags   u8   → has_tag / has_node
 21  2  node_len   u16                17  3  pad     ·    → round stride to 20
 23  2  tag_len    u16
 25  4  data_len   u32                seq → entry is a DIRECT seek:
 29  N  node                            (seq − start_seq) × 20
  .  M  tag                           No scan. No hashing.
  .  P  data+meta
  .  8  xxh3       u64

The inline ts in the index is a small but important trick: it lets the engine do a TTL binary-search ("which records have expired?") without ever touching the .data file, and the presence-flag bits answer "does this record have a tag?" equally cheaply. Rebuilding the in-memory index on restart is just a bulk read of the tiny .idx.

Sealed segments are immutable — so corruption is loud

The WAL's torn tail is an expected crash artifact, so the reader silently truncates it. A sealed segment is different: it is finished and never rewritten, so a checksum mismatch there is genuine bit-rot or corruption, not a torn write. The segment decoder surfaces the error rather than silently dropping data. Silent truncation of a sealed file would be a silent data loss — exactly what the design forbids.

Sealing

The active segment is buffered and appended to; it seals when it hits a size, event-count, or age cap, after which it is immutable and can be mmap'd for zero-copy reads. Sealed segments are the unit of tiering.

4.2 — Hot / cold tiers

Sealed segments live in a HOT tier (local, fast) and can be relocated to a COLD tier (a cheaper/larger store — a local folder today, object storage later) when configured via TOPICS_COLD_DIR. The rule that makes this safe:

Cold relocation is all-or-nothing, and never blocks the write path

A segment is fully present in COLD and fsynced before its HOT copy is removed — a crash mid-move leaves a usable copy in at least one tier (recovery reclaims the orphan). And a cold read — pulling an old segment back — never stalls or slows an active write or a live delivery. Cold is for history; the hot path stays hot.

05Metadata snapshots

Replaying the entire WAL from frame zero on every restart would be slow once the log is large. A snapshot is a periodic, self-contained capture of the engine's durable state plus a checkpoint position (wal_idx, wal_offset) that says "the WAL is fully reflected in this snapshot up to here — replay only what comes after."

5.1 — Format & what it carries

The file is a 20-byte header (magic "SNP1" + version + body_len + XXH3) wrapping a postcard-encoded body:

 Snapshot {
   id, ts, next_topic_id,
   checkpoint: { wal_idx, wal_offset, last_checkpoint_seq },
   topics:   [ SnapshotTopic  { name, topic_id, epoch, config,
                             base_seq, head_seq,
                             evict_floor, expiry_floor, delete_floor,
                             bytes_retained, live_count,
                             records: [ live, compacted, hole-free ] } ],
   routers: [ SnapshotRouter { …, forward_cursor, forwarded_total } ],
 }

The captured record set is the compacted form: deleted middle-holes and front-reclaimed prefixes are simply absent. Two capture rules are critical:

  • Memory-class topics are best-effort, not captured-empty. A memory topic takes the same group-committed WAL write + recovery path as disk, so its records may survive or be lost across a restart (no guarantee either way) — it is not reset-to-empty by contract, and head_seq never regresses above the acked head. The topic config always survives.
  • Never persist a staged-but-unpublished tail. A record can sit in the in-memory deque before head_seq advances (the WAL-first reservation). Capture clamps to seq ≤ head_seq so an invisible, not-yet-acked record is excluded — its WAL frame lands at/after the checkpoint and will be replayed instead.

The consistency ordering — why a snapshot loses nothing

Capture does things in an order chosen so no acked write can fall into the gap between "checkpoint position" and "materialized state":

  1. Flush barrier first. Write a durable CheckpointMark and wait for its fsync, so the writer's published position covers every prior committed frame. If that barrier fsync fails, the snapshot is abandoned — better no snapshot than one whose checkpoint races ahead of durability.
  2. Record the checkpoint position next. Any write committed after this read has a WAL offset ≥ the checkpoint, so recovery will replay it.
  3. Materialize each topic under its append lock, after draining in-flight publishes. This makes (head_seq, index) a consistent pair. A write that raced into both the snapshot and the post-checkpoint WAL is harmless: replay skips it by seq.

5.2 — The atomic swap

A snapshot must appear all-or-nothing to a restart that races with it. The write sequence:

 write body → snap.tmp
 fsync       snap.tmp            // contents durable BEFORE it has its real name
 rename      snap.tmp → snap-<id> // atomic: either old or new name resolves, never half
 fsync       directory           // the rename itself is now durable
 prune       older snapshots     // keep the newest valid one(s)

Recovery reads snapshots newest-first and skips any whose header magic, version, or XXH3 fails — a snapshot that was torn mid-write is simply ignored in favor of the previous good one (or, if none, a full WAL replay from zero). The atomic rename means a partially written snap.tmp never has a real snapshot name, so it is never even considered.

06Recovery — putting it back together

On startup the engine reconstructs its entire in-memory state from disk. The sequence is designed so that a crash during recovery itself is also safe (recovery is idempotent and re-runnable).

6.1 — Why replay is idempotent

The snapshot already contains state up to its checkpoint, but the checkpoint offset is conservative — some frames before it may already be reflected in the snapshot. So replay re-encounters records the snapshot already has. Two guards make that a no-op:

  • Seq-skip: an Append whose seq ≤ head_seq is already present; replay drops it. Replaying the same WAL twice yields the same state.
  • Contiguity guard: an Append whose seq ≠ head_seq + 1 is treated as a misdirected or torn frame and is not applied. A frame that somehow landed in the wrong place can never tear a gap into a topic.
Recovery is deterministic and re-runnable

Running recovery, crashing halfway, and running it again produces the same final state as running it once cleanly. There is no "half-recovered" disk state, because recovery only ever reads the WAL/segments and the single mutating step (torn-tail truncation) is itself idempotent — truncating to valid_len twice is the same as once.

07The filesystem seam

Every byte of durable I/O — WAL, segments, snapshots, directory fsyncs — goes through one narrow trait pair. This is what makes the durability guarantees testable rather than merely asserted.

 trait Fs : Send + Sync {                  trait File : Send {
   open(path, opts) -> File                  read_at(off, buf)
   rename(from, to)                          write_at(off, buf) -> usize   // short writes are first-class
   remove_file(path)                         set_len(n)
   read_dir(path)                            sync_data()               // fdatasync
   sync_dir(path)            sync_all()                // fsync
   create_dir_all(path)                      metadata_len()
   exists / metadata_len                     read_to_end_from(off)
 }                                         }

The signature detail that matters: write_at returns the number of bytes actually written. Short writes are a first-class outcome, not an error to paper over — the WAL's write_all_at_tail loops until the whole frame is on disk, exactly as a hostile disk that writes "3 of your 8 bytes" would force a real kernel to behave.

08The guarantees — theoretical & implied

Everything above exists to support a short list of promises. Here they are, stated plainly, with the mechanism that backs each.

G1 — Durability

An fsync-acked write survives any crash, including power loss. A disk-acked write survives a process crash and survives power loss except for an un-fsynced tail. A memory write makes no durability promise. Backed by: WAL group commit + the durability gate + tail-rewind-on-error.

G2 — No silent loss

If involuntary loss does happen (the un-fsynced tail of a disk-class topic, or an eviction under a cap/TTL), a reader is told, via an in-band tombstone gap signal. The system never quietly skips a seq a client was led to expect. Backed by: the evict_floor watermark and tombstones (§8.1).

G3 — Ordering & no gaps

Within a topic, seqs are dense and monotonic, and a reader observing seq N has necessarily been able to observe every seq below it (that wasn't explicitly deleted/evicted). Backed by: serialized seq assignment under the append lock + the ticket-gated publish order.

G4 — Crash atomicity of metadata

A snapshot is observed whole or not at all; a WAL batch persists whole or not at all; a cold relocation completes in both tiers or leaves a reclaimable orphan. No structure is ever observed half-applied. Backed by: tmp→fsync→rename→dir-fsync swaps and len-before tail rewind.

G5 — Corruption is explicit, never silent

A torn WAL tail is truncated (expected crash artifact). A checksum failure in a sealed segment or a snapshot is surfaced as an error, never silently dropped. Backed by: XXH3 on every frame + the immutable-means-loud rule.

G6 — Idempotent recovery & best-effort dedupe

Recovery is deterministic and re-runnable. Client idempotency keys suppress duplicate writes inside their window on the live path (the dedupe state is intentionally not persisted across restart — it is a best-effort live guard, not a durability promise). Backed by: seq-skip replay + the idempotency window.

8.1 — Watermarks & tombstones (the no-silent-loss machinery)

This is the most subtle and most important guarantee, so it gets its own anatomy. The engine tracks two kinds of "the floor moved up" — and they mean very different things to a reader.

WatermarkCauseVisible to reader?
evict_floor Involuntary loss — a cap or TTL forced records out before they were consumed. Yes — emits a tombstone. This is the only trigger for a tombstone.
delete_floor / earliest_seq Voluntary — a client explicitly deleted a prefix, or a node-filter skipped records. No — silent. The client asked for it; no surprise to signal.
Invariant — evict_floor ≤ earliest_seq

Involuntary loss can never get ahead of voluntary deletion. If a client already deleted up to seq 100, evicting up to seq 80 emits no tombstone (those records were already voluntarily gone). The tombstone only ever fires for the genuinely-surprising gap: data that vanished without the client asking.

A tombstone is an in-band, 200-level gap signal carrying a reason — Cap, Ttl, Mixed, Recreated, or FromSeqTooOld. A consumer reading the stream sees it inline and knows precisely what it missed and why. Contrast: deleting records yourself, or a multi-master node-filter dropping a loop, is silent — there is nothing to warn about.

09Proving it — the fault-injection harness

A durability claim is only as good as its test. The guarantees above are not asserted by hand — they are mechanically verified by driving the real engine through a crash at every persistence call site and checking the recovered state against a model.

The method, in four moves

  1. Inject a precise crash. The fail crate arms named failpoints at the exact call sites — wal::after_write, wal::after_fdatasync, snapshot::after_rename, recovery::before_truncate, and more. A test fires a crash at that instruction and not a microsecond elsewhere. (The feature is OFF by default — release builds compile these macros to nothing.)
  2. Crash a fake disk. FakeDisk.crash() discards every byte that wasn't fsynced, modelling power loss exactly.
  3. Recover and compare to a model. A pure-Rust model oracle (the analog of Postgres's wal_consistency_checking) tracks what should be recoverable. The check is a relaxed set relation:
   acked  ⊆  recovered  ⊆  model ∪ {acked}
   └────┬───┘            └──────┬─────────┘
   every write the         nothing appears that was never
   engine ACKED as         written — no resurrection, no
   durable IS there        invention of data
   after recovery

The relaxation is the whole point: a disk-class write that was not acked may or may not survive (it's allowed in the upper bound but not required in the lower), while an fsync-acked write must be in recovered. This captures the durability contract precisely.

The scale of it

15 model-oracle invariants (durability, no-silent-loss/no-gap, seq monotonicity, idempotent replay, torn-tail truncation, explicit corruption, atomic snapshot swap, segment idx⇒data consistency, all-or-nothing cold relocation, delete-never-resurrects, floors-preserved, lease projection, best-effort idempotency, visibility/publish atomicity, commit-token correctness) are checked against a 125-strategy crash-point catalog (the F-WAL-*, F-SNAP-*, F-SEG-*, F-COLD-*, F-RECLAIM-*, F-LEASE-*, F-REC-*, F-PUB-* … families), plus an always-on MonitorFs that asserts ordering on every test in the suite, plus property-based fuzzing.

This is why the guarantees can be stated as facts rather than hopes: the same engine that serves production is run, in-process, against a disk that lies, tears, fills up, and loses power at the worst possible instant — and the recovered state is held to the model every time.

The mantra

Remember this and you understand the whole engine

The WAL is the only truth. Acks are gated on it to the topic's durability class. Failed writes are rewound, never half-persisted. Segments and snapshots are rebuildable optimizations. Recovery is idempotent. Involuntary loss is always announced; voluntary loss is always silent. And every one of those sentences is enforced by a crash test against a hostile disk.
